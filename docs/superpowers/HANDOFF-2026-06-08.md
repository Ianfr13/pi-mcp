## OPEN — adversarial review must-fixes (NOT yet done)

Source: pi-mcp adversarial review **job `950262fd`**, runId `mq5ngqik-etoxpu` — full findings JSON in `/root/pi-mcp/.pi/workflows/runs/mq5ngqik-etoxpu.json` (`.result`). Triaged:

**Confirmed HIGH (fix these):**
- **Staleness vs timeout mismatch:** `agentTimeoutMs` is now 20 min but `config.StaleThreshold` is still 300 s (`config.go:24-26`) → `pi_status`/dashboard mark a healthy long-running agent `failed` at 5 min. Fix: raise StaleThreshold ≥ agent timeout (or add a running-job heartbeat / skip same-session live PIDs).
- **Dashboard `--state-dir` override builds `registry.json`** (`cmd/pi-dashboard/main.go:39-40`, help text :24) but canonical is `registry.db` → override = empty/split-brain dashboard. Fix: derive the override path with the `.db` helper.
- **Reconcile can prune a worktree while an orphaned `pi` child still writes it:** `runner.go` spawns via `exec.CommandContext` without `Setpgid`/`Setsid`; on abrupt server death the child is reparented and keeps writing; the next server's owner-scoped reconcile (dead PID) `Prune`s the worktree → data loss. Fix: before pruning a dead-owner write job, check `rec.PID` alive → skip or terminate-and-wait. (Consider `Setpgid` so cancel kills the whole group.)
- **`Registry.Close()` closes the store before job goroutines finish** (`registry.go:488-509` vs `runAttempts`/`finish` flush) → terminal flush can hit a closed `*sql.DB` and be lost (mitigated: next reconcile reterminalizes). Fix: `sync.WaitGroup` for job goroutines; wait before `store.Close()`; block queue promotion during Close.

**Valid hardening (MED/LOW):**
- Owner-guard the UPSERT conflict clause (`WHERE ownerPid=me OR ownerPid=0`) so a future caller can't clobber a foreign live row.
- Run `Reconcile` periodically (today only at boot → dead siblings linger until some server restarts).
- Use `ownerStartedAt` (stored but unused) as a PID-reuse tiebreak (e.g. `/proc/<pid>` start-time) so a recycled PID doesn't strand dead jobs.
- Forbid per-agent `timeoutMs` (and finite phase budgets) in the forcing prompt — an orchestrator can still emit `agent(...,{timeoutMs:300000})` and recreate the 5-min kill.
- Dashboard `ReadRegistry` opens a new `*sql.DB` every poll (1 Hz) → use one read-only handle for the poller lifetime.
- Re-add spec-promised pragmas dropped in code: writer `foreign_keys(ON)`, dashboard `query_only(1)`.
- Migration atomicity: quarantine corrupt legacy JSON, treat already-renamed concurrent races as success.

**Disputed / won't-fix (documented):**
- "Finite tokenBudget regresses unlimited" — `2e9` is practically unlimited; we set it *because* gpt-5.5 disobeys "omit it" and sets a small one. Keep (at most: also instruct omit).
- "Migration loses rolling-upgrade writes" — that's the known split-brain; resolved by an atomic cutover (reopen all sessions). Operational, not a code bug.
- "pi_status cross-server" — intentionally out of scope (in-memory own-jobs path).
